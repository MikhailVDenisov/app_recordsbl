# ВМ в Cloud.ru Evolution для API RecordsBL

Эта ВМ поднимает **`server/`** (Node.js + Fastify) так, чтобы он мог подключаться к **Managed PostgreSQL® по внутреннему IP** из той же подсети. Мобильное приложение ходит на API по **публичному адресу** ВМ (или домену).

Официальный мастер создания ВМ: [Создать виртуальную машину в облаке Evolution](https://cloud.ru/docs/virtual-machines/ug/topics/guides__create-vm). Ниже — параметры, согласованные с этой схемой.

### Публичные образы или Маркетплейс?

| Источник | Что выбрать для RecordsBL |
|----------|---------------------------|
| **Публичные** | **Предпочтительно:** **Ubuntu 22.04/24.04 LTS** без доп. ПО — «чистый» сервер, ставите Node.js сами ([NodeSource](https://github.com/nodesource/distributions), nvm и т.д.). Совпадает с разделом ниже. |
| **Маркетплейс → LAMP** | **Не подходит:** Apache + **MySQL** + PHP. БД у вас — **Managed PostgreSQL**, не MySQL на ВМ; лишние сервисы только мешают. |
| **Маркетплейс → Node.js** | Допустимо, если в карточке образа указан **Node 20+ LTS** и устраивает способ обновлений; проверьте актуальность версии. |
| **Маркетплейс → Docker** | Имеет смысл, только если планируете **контейнер** с API; для первого деплоя проще ВМ + systemd без Docker. |
| **CentOS / Debian (публичные)** | Рабочие варианты; для Node чаще удобнее **Ubuntu** из‑за документации и примеров. |

## Обязательные условия по сети

| Параметр | Значение |
|----------|----------|
| **Проект** | Тот же, что у кластера Managed PostgreSQL. |
| **VPC и подсеть** | **Те же**, что у кластера PostgreSQL (иначе внутренний адрес БД будет недоступен). |
| **Зона доступности** | Согласована с подсетью (часто одна зона на подсеть). |
| **Публичный IP** | Нужен клиентам (приложению) для HTTPS к API; в мастере ВМ включите выдачу публичного адреса или привяжите плавающий IP. |

## Рекомендуемые параметры ВМ

Точные названия тарифов в консоли могут меняться — ориентируйтесь на класс **General purpose** / аналог «универсальной» ВМ.

| Параметр | Рекомендация для старта | Комментарий |
|----------|-------------------------|-------------|
| **ОС (образ)** | **Ubuntu 24.04 LTS** или **22.04 LTS** | Удобны для Node.js, много гайдов; в консоли: публичные образы Linux. |
| **vCPU** | **2** | Для небольшой нагрузки API достаточно; при росте — увеличить. |
| **RAM** | **4 ГБ** | Минимум комфортный для Node + ОС; при узком бюджете можно **2 ГБ** и мониторить swap. |
| **Диск** | **20–40 ГБ** SSD / сетевой диск | ОС, `node_modules`, логи; при больших логах — больше. |

Итог: типичная стартовая конфигурация — **2 vCPU / 4 ГБ RAM / 30 ГБ диск**, Ubuntu 22.04 или 24.04.

## Группы безопасности (фаервол)

Настройте правила **для ВМ** и **для кластера PostgreSQL** (имена в UI могут отличаться):

**На ВМ (входящий трафик):**

- **TCP 22** — SSH (лучше не с `0.0.0.0/0`, а с вашего офисного/VPN IP).
- **TCP 443** — HTTPS (после настройки reverse proxy; либо временно **3000** только для теста, закрыть потом).
- **TCP 80** — опционально, для выдачи сертификата Let’s Encrypt (HTTP-01).

**Исходящий с ВМ:** разрешить доступ в интернет (Object Storage `https://s3.cloud.ru`, обновления пакетов) и **к внутреннему адресу PostgreSQL** (часто достаточно правил «внутри VPC» по документации Cloud.ru).

**На кластере PostgreSQL:** разрешить **входящий порт БД** (обычно **5432**) **с внутреннего IP или security group этой ВМ**, а не из интернета.

## Что дальше, если ВМ уже есть (Ubuntu 22.04 + публичный IP)

1. **Группы безопасности** — открыть **SSH 22** к ВМ (хотя бы с вашего IP), иначе по публичному адресу не зайдёте. Позже — **443** (и при необходимости **80**) для HTTPS.
2. **SSH:** `ssh ubuntu@<публичный_IP>` (логин может быть `ubuntu` или другой — см. подсказку в консоли Cloud.ru).
3. **Обновления:** `sudo apt update && sudo apt upgrade -y`
4. **Node.js 20 LTS** — по [инструкции NodeSource](https://github.com/nodesource/distributions#installation-instructions) для Ubuntu или через `nvm`.
5. **Git** (если клонируете репозиторий): `sudo apt install -y git`
6. **Код:** клонировать монорепозиторий или скопировать только каталог `server/` на ВМ, перейти в `server/`.
7. **Конфиг:** `cp .env.example .env`, заполнить **`DATABASE_URL`** (внутренний хост PostgreSQL из консоли), **`S3_*`**, **`PORT=3000`**.
8. **Сборка и миграции:** `npm ci && npm run build && npm run db:migrate`
9. **Автозапуск:** unit **systemd** или **pm2** для `node dist/index.js` (слушает `0.0.0.0`, см. `server/src/index.ts`).
10. **HTTPS снаружи:** **nginx** или **Caddy** на 443 → прокси на `http://127.0.0.1:3000`. Для Let’s Encrypt удобнее **домен** (на A-запись на публичный IP); чистый IP без домена с сертификатом сложнее.
11. **В приложении** в настройках указать базовый URL API: `https://ваш-домен` (или временно `http://IP:3000` только для теста, не для продакшена).

## После создания ВМ: ПО на сервере

1. Подключение по SSH (логин зависит от образа; для Ubuntu часто `ubuntu` или тот, что указан в консоли).

2. Установите **Node.js 20 LTS** (или новее в рамках поддержки проекта), например через [NodeSource](https://github.com/nodesource/distributions) или пакеты дистрибутива.

3. Скопируйте репозиторий или артефакт деплоя, в каталоге `server/`:

   ```bash
   cp .env.example .env
   # заполните DATABASE_URL (внутренний хост БД), S3_*, PORT
   npm ci
   npm run build   # если в проекте есть build; иначе npm install
   npm run db:migrate
   ```

4. Запуск процесса: **systemd** (юнит для `node dist/index.js` или `node --import tsx`) или **pm2** — чтобы API переживал перезагрузку.

5. **HTTPS:** поставьте **nginx** или **Caddy** перед Node, проксируйте на `127.0.0.1:3000` (или ваш `PORT`), получите сертификат (Let’s Encrypt). В мобильном приложении в настройках укажите **`https://ваш-домен-или-ip`** (для продакшена лучше домен с валидным TLS).

## SSH: `Permission denied (publickey)`

Значит на ВМ включена только аутентификация по ключу, а ваш Mac не предложил **подходящий приватный ключ**.

1. При создании ВМ в Cloud.ru обычно указывают **SSH-ключ** (или создают пару в консоли). Подключаться нужно с **тем же приватным ключом**, что соответствует публичному ключу, прописанному для этой ВМ:
   ```bash
   ssh -i ~/.ssh/ИМЯ_ВАШЕГО_КЛЮЧА admin_aist_app_vm@ПУБЛИЧНЫЙ_IP
   ```
   Если ключ не в стандартных `id_rsa` / `id_ed25519`, без `-i` клиент его не найдёт.

2. Права на файл ключа: `chmod 600 ~/.ssh/ИМЯ_КЛЮЧА`

3. Если ключ потерян или к ВМ **не был привязан ни один ключ** — зайдите в **консоль Cloud.ru**: добавьте свой **публичный** ключ к ВМ (если платформа это позволяет) или используйте **веб-консоль / serial console** к ВМ по документации Cloud.ru и пропишите в `~/.ssh/authorized_keys` на сервере строку из `~/.ssh/id_ed25519.pub` с Mac.

4. Логин `admin_aist_app_vm` должен совпадать с тем, что задан для образа в консоли; для чистого Ubuntu часто `ubuntu`. Уточните в карточке ВМ или в письме/мастере создания.

## Проверка

- С ВМ: `curl -sS http://127.0.0.1:3000/health` (или тот путь, который реализован в API).
- С ноутбука: `curl https://ваш-публичный-адрес/...`
- БД: на ВМ `psql "$DATABASE_URL" -c 'SELECT 1'` с тем же `DATABASE_URL`, что у приложения.
