# Managed PostgreSQL в Cloud.ru

Пошаговая инструкция для кластера БД, сети и подключения к нему из API (`server/`). Названия пунктов меню в [личном кабинете Cloud.ru](https://console.cloud.ru/) могут слегка отличаться — ориентируйтесь на [документацию Managed PostgreSQL](https://cloud.ru/docs/paas-postgresql/).

Архитектурно это тот же **PostgreSQL + строка `DATABASE_URL`** для бэкенда; у Managed PostgreSQL® в Evolution **нет прямого доступа к кластеру из интернета** — см. раздел ниже.

## Как обеспечить запись в БД (рекомендуемая схема)

По [ограничениям Cloud.ru](https://cloud.ru/docs/paas-postgresql/) кластер Managed PostgreSQL® доступен **только по внутреннему IP** с **виртуальной машины в Evolution**, причём ВМ и кластер должны быть в **одном проекте и одной подсети**. К ВМ при этом обычно вешают **публичный IP** (или балансировщик).

Цепочка данных такая:

1. **Мобильное приложение** не подключается к PostgreSQL. Оно ходит по **HTTPS** на ваш **API** (`server/`) — на публичный адрес или домен, который указывает на ВМ (или на балансировщик перед ВМ).
2. **API на ВМ** в той же VPC, что и БД, подключается к PostgreSQL по **`DATABASE_URL` с внутренним хостом** (private hostname / внутренний IP из консоли). Запись в таблицы делает **только этот сервер**.
3. **Миграции** (`npm run db:migrate`) выполняйте **на той же ВМ** (по SSH), в CI после деплоя на ВМ, либо через временный **SSH-туннель** с ноутбука на ВМ, если документация это допускает.

Итог: «запись в БД» для пользователей приложения — это **запись через ваш REST API**; прямой доступ к PostgreSQL с телефона или домашнего ПК к кластеру **не требуется** и при описанной модели **недоступен**.

**ВМ под API в Evolution:** ОС, размеры, сеть, фаервол и установка Node — в отдельном файле **[vm-api-evolution-cloud-ru.md](vm-api-evolution-cloud-ru.md)**.

## 1. Подготовка

1. Войдите в [консоль Cloud.ru](https://console.cloud.ru/).
2. Создайте **проект** / выберите каталог, в котором будут ресурсы.
3. Убедитесь, что сервис **Managed PostgreSQL** доступен по тарифу и квотам.

## 2. Сеть (VPC)

Кластер БД должен быть в **подсети** внутри VPC. Создайте сеть/подсеть по мастеру в консоли, если их ещё нет; CIDR подсетей не должны пересекаться.

## 3. Создание кластера PostgreSQL

1. Откройте **Managed PostgreSQL** (или **RDS for PostgreSQL**, если используете этот продукт — см. [документацию](https://cloud.ru/docs/paas-postgresql/)) → **Создать** экземпляр / кластер.
2. Задайте **имя**, версию **PostgreSQL** 15 или 16 (совместимо с проектом).
3. Выберите **класс хоста** и **диск** под нагрузку.
4. Укажите **VPC и подсеть** (и зону доступности) — **ту же подсеть**, в которой будет ВМ с API (см. раздел «Как обеспечить запись в БД»).
5. Подключение к кластеру — **только из внутренней сети Evolution** (внутренний адрес в карточке кластера). Публичного доступа к PostgreSQL у продукта может не быть — это нормально: API размещается на ВМ в этой же сети.
6. Задайте **пользователя БД** и **надёжный пароль**.
7. Создайте **базу данных** (например `recordsbl`).

Дождитесь статуса готовности кластера.

## 4. Виртуальная машина с API и доступ к БД

1. Создайте **ВМ в Evolution** в **том же проекте и подсети**, что и кластер PostgreSQL. Назначьте ВМ **публичный IP** (или используйте балансировщик с публичным адресом) — к нему будут обращаться клиенты по HTTPS.
2. Установите на ВМ Node.js, задеплойте `server/`, задайте **`DATABASE_URL` с внутренним хостом** кластера (из карточки БД в консоли).
3. В **группах безопасности** разрешите входящий **5432** (или порт из документации) **с адреса ВМ** / из подсети API на кластер БД; наружу порт PostgreSQL открывать не обязательно.
4. Для SSL к БД следуйте [инструкции по подключению](https://cloud.ru/docs/paas-postgresql/ug/topics/guides__databases__connecting-to-database.html).

Проверка `psql` и миграций с **ноутбука**: либо **SSH на ВМ** и запуск команд там, либо туннель, если используете его по гайду Cloud.ru.

## 5. Строка подключения `DATABASE_URL`

Формат для Node.js и `psql`:

```text
postgres://USER:PASSWORD@HOST:PORT/DBNAME?sslmode=require
```

- **HOST** и **PORT** — из консоли (часто **5432** или отдельный порт для SSL — смотрите в интерфейсе).
- Пароль в URL **URL-кодируйте**, если есть `@`, `#`, `%` и т.д.

Пример в `server/.env` на **ВМ** (подставьте **внутренний** хост из консоли, не публичный):

```env
DATABASE_URL=postgres://records:YOUR_PASSWORD@10.x.x.x:5432/recordsbl?sslmode=require
```

## 6. Применение схемы в БД

```bash
cd server
cp .env.example .env
# вставьте DATABASE_URL
npm install
npm run db:migrate
```

Вручную:

```bash
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -f server/sql/001_init.sql
```

## 7. Проверка

```bash
export DATABASE_URL='...'
./scripts/yc-check-postgres.sh
```

Или:

```bash
psql "$DATABASE_URL" -c "SELECT count(*) FROM meetings;"
```

## 8. Частые проблемы

| Симптом | Что сделать |
|--------|-------------|
| `timeout` / `connection refused` | API и БД в одной подсети; SG разрешают трафик ВМ→кластер; `DATABASE_URL` с **внутренним** хостом. |
| SSL | Режим `sslmode=require`, при необходимости CA из документации Cloud.ru. |
| `password authentication failed` | Пользователь и пароль; экранирование в URL. |
| `CREATE EXTENSION` | Расширение `uuid-ossp` обычно доступно; иначе — роль владельца БД. |
